UPI Fraud Detection

Complete Beginner's Guide — Everything explained from zero in plain English. Read this before your interview.

1. What is this project and why does it exist?

The real-world problem

Every time you pay on PhonePe, GPay, or Paytm — a transaction happens. Most are legitimate. But some are fraud — someone stole your UPI PIN, cloned your SIM, or tricked you into sending money.

Banks process millions of transactions per second. No human team can check each one. So companies build machine learning models that automatically scan every transaction and raise a flag if it looks suspicious.

Real World Analogy
Think of it like a metal detector at an airport. Every passenger goes through it. The detector doesn't know who is a criminal — it just knows the pattern of things criminals carry. Our model doesn't know who is a fraudster — it knows the pattern of fraudulent transactions (unusual amount, new device, late night, etc.).

What is Machine Learning?

Traditional programming: You write rules. "If amount > 50,000 AND night time → flag as fraud."

Problem: Fraudsters evolve. They learn your rules and work around them.

Machine Learning: Instead of writing rules yourself, you show the computer thousands of examples of fraud and non-fraud. The computer finds the patterns automatically — including ones humans can't see. This is called training a model.

What is our project specifically?

We built a binary classification system — every transaction gets classified into one of two classes:

Class 0 = Legitimate transaction ✅
Class 1 = Fraudulent transaction ❌

We trained the model on 284,807 real transactions. Then we test it on new transactions it has never seen. If it correctly identifies fraud — it works.

3. Dataset — every detail

The Kaggle Credit Card Fraud Dataset

Source: Real transactions from European cardholders in September 2013.
Collected by: Université Libre de Bruxelles (ULB) — a university research group in Belgium.
Why it's trusted: Used in hundreds of published research papers. Real bank data.
Size: 284,807 transactions | Fraud: 492 (0.17%) | File size: 67.4 MB

What are the columns?

V1 to V28: 28 anonymized features. The bank ran PCA (Principal Component Analysis) on original data — things like merchant name, location, device ID — to hide sensitive information. Patterns preserved, identities hidden.

Time: Seconds since the first transaction. Not a clock — just a counter.

Amount: Transaction amount in Euros. Raw, unscaled.

Class: Target column. 0 = legitimate, 1 = fraud. This is what we predict.

💡 Why anonymized? Real transaction data contains personal information — names, addresses. Banks legally cannot share this. PCA transforms data into abstract numbers that preserve patterns without revealing identities. Standard in financial industry.

The class imbalance problem

Out of 284,807 transactions — only 492 are fraud. That is 0.17%. This is called class imbalance.

Total transactions: 284,807 Legitimate: 284,315 (99.83%) Fraud: 492 (0.17%) Imbalance ratio: 578 legitimate for every 1 fraud
🚨 A "dumb" model predicting everything as legitimate gets 99.83% accuracy. This is why accuracy is USELESS for fraud detection. We use F1, Recall, and AUC-ROC instead.

"But this is credit card data, not UPI data?"

Honest answer for interviews: "Real UPI fraud datasets are proprietary — companies like Razorpay and PhonePe don't release them publicly. This is the most widely used benchmark fraud dataset in ML research. The features represent anonymized behavioral transaction signals, which are similar to what UPI fraud detection systems use. The pipeline, techniques, and deployment approach are completely applicable to real UPI systems."

✅ This answer is honest, shows domain knowledge, and explains why the choice is legitimate. Never lie about your dataset — explain your reasoning confidently.

5. Feature Engineering

What is feature engineering?

The raw dataset has columns as collected. But sometimes raw columns aren't the most useful form for a model. Feature engineering is creating new columns from existing ones that better capture the patterns you care about.

Analogy
Raw potatoes exist — but you peel them, cut them, season them to make them more useful in a recipe. Feature engineering prepares raw data for the model to use most effectively.

New feature: hour

Raw Time column = "seconds since first transaction" — like 47382. Meaningless for detecting night fraud. We converted it:

hour = (Time // 3600) % 24 Example: 47382 seconds → hour 13 (1pm) → low risk Example: 10800 seconds → hour 3 (3am) → high risk

Now the model can learn "transactions at 3am have higher fraud rate than 2pm."

New feature: is_night

From hour, we created a binary flag: 1 if between 10pm and 6am, else 0.

is_night = 1 if (hour < 6) or (hour >= 22) else 0

Direct encoding of the risky time window as a single 0/1 value. Very clean signal for the model.

New feature: Amount_scaled

Raw Amount values range from €0.01 to €25,691. V1-V28 are already scaled (range -5 to +5). If Amount stays raw, models that use magnitude will wrongly think Amount is the most important feature just because its numbers are bigger.

StandardScaler transforms Amount to have mean=0 and standard deviation=1. Now all features are on the same scale.

Before: Amount = [0.0, 25691.16, 1.00, 150.00] After: Amount = [-0.34, 10.52, -0.34, -0.28]
⚠️ Critical: Fit the scaler ONLY on training data. If you fit on all data, test set statistics leak into training — data leakage.

6. Handling Class Imbalance

Train/Test Split

Training set (80%): 227,846 rows — used to teach the model
Test set (20%): 56,961 rows — used to evaluate how well it learned

The model never sees the test set during training. Like an exam — you study with practice questions, get tested on completely new questions.

X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.2, # 20% goes to test random_state=42, # same split every time you run stratify=y # same fraud ratio in both sets )

Why stratify=y?

With only 492 fraud cases total, a random split might put 450 frauds in training and only 42 in test — by pure chance. Then our evaluation would be unreliable.

stratify=y guarantees both sets maintain the exact same fraud ratio.

Result: Train = 0.1729% fraud, Test = 0.1720% fraud — almost identical ✅

SMOTE — fixing the imbalance

After splitting, training has 394 real fraud cases vs 227,452 legitimate. SMOTE fixes this by creating synthetic fraud examples.

How SMOTE works step by step:
1. Pick a real fraud transaction (Point A)
2. Find its 5 nearest fraud neighbours
3. Pick one randomly (Point B)
4. Create a new point somewhere between A and B
5. Repeat until fraud count matches legitimate count

Analogy
You have 5 cat photos and 1000 dog photos. A model trained on this won't learn cats well. SMOTE is like an artist who studies your 5 cat photos and draws 995 new realistic cat variations. Now 1000 cats and 1000 dogs — balanced training.
Before SMOTE: Legitimate=227,452 Fraud=394 After SMOTE: Legitimate=227,452 Fraud=227,451 (balanced!) Test set: UNTOUCHED — still 0.17% fraud (real world)
⚠️ NEVER apply SMOTE to test data. Test must reflect the real world. If you balance the test set, your scores look amazing but mean nothing in production.

7. The 4 Models — what they are and how they work

Model 1: Logistic Regression — the baseline

Despite the name, Logistic Regression is a classification algorithm. It draws a straight line (in many dimensions) that separates fraud from legitimate. It multiplies each feature by a learned weight, adds them up, and passes through a function that gives a probability between 0 and 1.

Analogy
Draws one straight line through a crowd to separate criminals from non-criminals. Works when separation is simple. Real fraud doesn't follow a straight line — patterns are complex and non-linear. That's why we need more powerful models.
Results: AUC=0.9721 F1=0.1017 Recall=0.9082 Precision=0.0539 Problem: High recall (catches 91% fraud) BUT terrible precision Only 5% of its fraud flags are real fraud → It flags almost EVERYTHING as fraud to play safe

Model 2: Decision Tree

Asks a series of yes/no questions about a transaction to reach a verdict — like a flowchart. Each question is chosen to best separate fraud from legitimate. Keeps splitting until confident.

Analogy
Like the game 20 Questions. "Is it an animal? Bigger than a dog? Has stripes?" Each question narrows the answer. A Decision Tree plays 20 Questions with transaction features to reach fraud/not-fraud verdict.
Results: AUC=0.8869 F1=0.1202 Recall=0.8367 Problem: Lowest AUC — single trees overfit to training data

Model 3: Random Forest

Builds 100 decision trees, each trained on a random subset of data and features. Every tree is different. New transaction → ask all 100 trees → majority vote wins. This is called bagging.

Analogy
Don't trust one judge — assemble a jury of 100. Each juror saw different evidence. Their collective vote is far more reliable than any single opinion.
Results: AUC=0.9836 F1=0.5139 Recall=0.8500 Much better than single Decision Tree! Highest AUC of all 4 models

Model 4: XGBoost — our best model

Builds 200 trees sequentially. Each new tree is built specifically to fix the mistakes of all previous trees. This is called gradient boosting.

Step 1: Build Tree 1. Check mistakes.
Step 2: Build Tree 2 to fix Tree 1's mistakes.
Step 3: Build Tree 3 to fix what Tree 1+2 still got wrong.
... repeat 200 times.
Final prediction = sum of all 200 trees.

Analogy
A student takes a test, gets some wrong. Teacher shows which ones. Student studies specifically those and retakes. Gets new ones wrong. Studies those. After 200 rounds of targeted study, the student is extremely good at exactly the hard questions.
Results: AUC=0.9819 F1=0.6391 Recall=0.8700 Precision=0.5100 🏆 Best F1 — best balance of precision and recall Catches 87% of all frauds For every 2 flags: 1 real fraud, 1 false alarm Acceptable in fraud detection

Random Forest vs XGBoost — the key difference

Random Forest (Bagging): Trees built in parallel, independently. They don't know about each other. Average their answers. Reduces overfitting (variance).

XGBoost (Boosting): Trees built sequentially. Each knows what previous ones got wrong. Fixes errors step by step. Reduces both overfitting and underfitting.

Why XGBoost wins: Sequential error correction handles complex non-linear fraud patterns better. Random Forest AUC=0.9836 but F1=0.51. XGBoost AUC=0.9819 but F1=0.64. Better real-world performance.

8. Evaluation Metrics — how we measure performance

Why accuracy fails

With 0.17% fraud — a model predicting "not fraud" for everything gets 99.83% accuracy. Looks incredible. Catches zero fraud. Completely useless.

Dumb model (always says "legitimate"): Accuracy = 99.83% ← looks great! Fraud caught = 0 out of 492 ← completely useless XGBoost: Accuracy = 99.94% ← barely better! Fraud caught = 428 out of 492 ← actually useful!

Confusion Matrix — the foundation of everything

Predicted: Legit Predicted: Fraud Actual: Legit TN ✅ FP ❌ (false alarm) Actual: Fraud FN ❌ (missed!) TP ✅ (caught!) TN = Legitimate, correctly identified as legitimate TP = Fraud, correctly identified as fraud ← want HIGH FP = Legitimate, wrongly flagged as fraud ← false alarm FN = Fraud, missed (called legitimate) ← MOST COSTLY
In fraud detection, False Negatives are the worst. A missed fraud = real financial loss. False Positives are annoying (blocked transaction) but don't cause direct financial loss.

Precision and Recall

Precision = of all transactions we flagged as fraud, how many actually were fraud?
Formula: TP / (TP + FP)
XGBoost Precision = 0.51 → 51% of fraud flags are real fraud. 49% are false alarms.

Recall = of all real frauds that existed, how many did we catch?
Formula: TP / (TP + FN)
XGBoost Recall = 0.87 → We caught 87% of all real frauds. Missed 13%.

Analogy
Precision: You arrest 100 people. 51 are actually criminals. Precision = 51%.
Recall: There are 100 criminals in the city. You caught 87. Recall = 87%.
In fraud: Recall matters more than Precision. Missing a fraud = bank loses money. Low precision just means more customer service calls — manageable. We tune the threshold to favor recall.

F1 Score

F1 = harmonic mean of Precision and Recall. Single number that balances both.

Formula: F1 = 2 × (Precision × Recall) / (Precision + Recall)

Why harmonic mean? If one value is very low, F1 gets dragged down hard. A model with Precision=1.0 and Recall=0.01 gets F1=0.02, not 0.505. This punishes extreme imbalance between precision and recall — forces you to have both reasonably good.

XGBoost: Precision=0.51, Recall=0.87 F1 = 2 × (0.51 × 0.87) / (0.51 + 0.87) = 0.64

AUC-ROC

ROC curve plots True Positive Rate (Recall) on Y-axis vs False Positive Rate on X-axis, at every possible threshold from 0 to 1. AUC = Area Under that curve.

1.0 = perfect model
0.5 = random guessing (flip a coin)
Our XGBoost = 0.9819 — excellent

Why use it? It's threshold-independent — tells you how fundamentally good the model is regardless of where you set the fraud cutoff. Best metric for comparing models.

Threshold Tuning

Default: if fraud probability > 0.5, call it fraud. But 0.5 is arbitrary.

Lower threshold (0.3): More transactions flagged → Recall goes up → Precision goes down
Higher threshold (0.7): Fewer transactions flagged → Precision goes up → Recall goes down

We plotted F1, Precision, Recall at every threshold from 0.1 to 0.9 and found the best one.

✅ Key interview insight: "The threshold is a business parameter, not just a technical one. A bank losing ₹10 crore per fraud incident will set a very low threshold — maximize recall even with more false alarms."

9. SHAP — Explaining the model's decisions

Why explainability matters

Imagine the bank's model blocks your transaction. You call support and ask why. "The model said so" is not acceptable — legally or practically.

The bank needs to say: "Your transaction was flagged because it was 8x your usual amount, your device was new, and it was at 3am." That's an explanation.

In India, RBI guidelines require financial institutions to explain automated decisions. SHAP is how we produce those explanations.

What is SHAP — simple explanation

SHAP = SHapley Additive exPlanations. Based on Shapley values from game theory (Nobel Prize concept).

For each prediction, SHAP gives every feature a score:
Positive score = feature pushed prediction toward FRAUD
Negative score = feature pushed prediction toward LEGITIMATE
Zero = feature had no effect on this prediction

Analogy
4 friends start a business and make ₹1,00,000 profit. How do you fairly split it? Equal split ignores that some worked harder. Shapley values calculate each person's fair contribution by checking all possible team combinations. SHAP does this for features — fairly credits each one for its contribution to the prediction.

Global SHAP — which features matter overall?

The summary plot shows all features ranked by importance across all test transactions. Each dot = one transaction. Red = high feature value. Blue = low feature value. Position left/right = pushed toward legitimate/fraud.

This tells us which features the model relies on most overall. The top features are what you talk about in interviews.

Local SHAP — why was THIS specific transaction flagged?

We picked one fraud transaction and explained exactly why it was flagged. The chart shows each feature's contribution for that one transaction. Red bars = pushed toward fraud. Blue = toward legitimate. Their sum = the final fraud score.

✅ This is "interview gold." You can say: "Feature V14 had a very low value for this transaction, strongly indicating fraud. V17 also contributed. Amount was above average. Together they pushed the fraud probability to 94%." Most candidates can't do this — SHAP sets you apart.

Saving the model with joblib

Training takes time. We save the trained model to a file so we don't retrain every time.

joblib.dump(xgb, 'model/fraud_model.pkl') # model joblib.dump(scaler, 'model/scaler.pkl') # scaler joblib.dump(explainer, 'model/shap_explainer.pkl') # SHAP
💡 Why save the scaler separately? New transactions must be scaled the exact same way as training data. The saved scaler remembers the mean and std from training. Creating a new scaler would give different scaling — predictions would be wrong.

Why XGBoost over Random Forest?
Random Forest uses bagging — builds many trees independently, each on a random data subset. They vote. Trees don't know about each other.

XGBoost uses gradient boosting — builds trees sequentially. Each new tree is trained specifically on the mistakes of all previous trees. Like a student who studies exactly the questions they got wrong after each test.

What are SHAP values?
SHAP = SHapley Additive exPlanations — based on Shapley values from cooperative game theory. For every single prediction, SHAP tells you exactly how much each feature contributed. Positive SHAP = feature pushed toward fraud. Negative SHAP = pushed toward legitimate.

For a flagged transaction I can say: "V14 was very low (-4.2), strongly indicating fraud (+0.8 contribution). V17 also contributed (+0.6). Amount was above average (+0.3). Together these pushed fraud probability to 94%."

This is essential in banking — regulators require an explanation for every declined transaction. Black-box models aren't acceptable.
hot — asked in every interview
What is data leakage?
Data leakage = information from outside your training set sneaking in and making evaluation unrealistically good. Model looks great, fails in production.

I avoided it 3 ways:
1. SMOTE only on training data — test reflects real world
2. StandardScaler fitted on training data only — then applied to test
3. Stratified split — honest fraud distribution

If I had fitted the scaler on ALL data before splitting, the scaler would know the test set's statistics. That's leakage — model indirectly "sees" test data during training.
